# Codex 图像生成排查交接

日期：2026-07-21

## 当前结论

本机 Codex 的图像生成失败，根因已经定位到网络代理的 TLS 证书链，而不是提示词、图片文件、Git 仓库或 ChatGPT 账号权限。

- ChatGPT 网页端可以正常登录并生成图片，说明账号的图像权限和服务本身正常。
- 多次调用 Codex 的 `image_gen` 均在发送请求阶段失败，错误为：
  `network error: error sending request for url (https://chatgpt.com/backend-api/codex/images/generations)`。
- `codex doctor` 显示安装、配置解析和 ChatGPT 登录令牌均正常；版本为 `0.144.6`。
- 它同时明确报告：
  - Responses WebSocket：`invalid peer certificate: UnknownIssuer`
  - ChatGPT 必需 API 端点无法通过 HTTP 访问。
- 所有大小写形式的 `HTTP_PROXY`、`HTTPS_PROXY`、`ALL_PROXY` 都指向 `http://127.0.0.1:7897`。
- 经代理访问 `chatgpt.com` 时，Windows Schannel 报“没有可用的凭据”；绕过代理后，直连在 10 秒内超时。
- 下载 Codex 官方手册时同样经代理收到 HTTP 403。这个失败和图像生成走的是同一类外部 HTTPS 路径。

因此，“重新安装/更新 Codex 导致图像功能缺失”的可能性很低。更符合证据的解释是：本地代理在转发 Codex/ChatGPT 请求时使用了 Codex 不信任的证书，或该代理的 HTTPS CONNECT/证书配置异常；而当前网络又不能直接访问服务。

另有两条非阻塞警告：Codex 无法清理 `C:\Users\huawei\.codex\tmp\arg0` 中的旧临时目录，也无法创建 PATH 别名。这两条与图像请求在网络发送阶段失败没有直接因果证据，暂未做任何删除或权限修改。

## 建议的下一步

1. 在代理软件中确认 `127.0.0.1:7897` 当前运行、可代理 HTTPS，并为 Windows/Codex 正确安装和信任其根证书；不要通过关闭 TLS 验证来绕过问题。
2. 如代理有“HTTPS 解密/MITM”选项，优先让 `chatgpt.com` 不被解密，或修复该解密所用根证书的受信任状态。
3. 重新启动代理和 Codex 后，在终端运行 `codex doctor`。只有 `reachability` 不再失败后，再尝试一次极短的 `image_gen` 提示词。
4. 如果代理软件无法修复，改用一个能正常访问 ChatGPT API、且证书链受 Windows 信任的代理。直连当前会超时，单纯清空代理变量不会解决问题。
